# Deep Web Search 产品文档（V0.8）

## 1. 产品定位

Deep Web Search 不是传统的网页搜索接口增强版，而是一个面向大模型的“类人研究工作流系统”。

它的目标不是让模型搜到更多链接，而是让模型具备下面这套能力：

- 先理解任务，再决定怎么搜。
- 不只搜网页，也搜视频平台、社媒、论坛和资讯站。
- 先收集候选内容的元数据，再判断哪些值得深读。
- 对高价值视频、音频、文章、帖子做深度提取和正文化。
- 在回答前做交叉验证、反思和证据组织。

一句话定义：

> 让大模型像研究员一样规划搜索、调用工具、读取多模态内容、验证事实，并输出证据化结论。

## 2. 为什么现有 Web Search 不够

当前很多大模型的 Web Search 仍然停留在“输入 query -> 返回网页列表 -> 拼接摘要”的阶段，主要问题是：

- 搜索入口单一，主要依赖网页和搜索引擎。
- 对抖音、哔哩哔哩、YouTube、微博、小红书、论坛等平台利用不足。
- 缺少任务拆解能力，用户一提问就直接搜。
- 缺少候选筛选层，无法先看标题、作者、时间、热度再决定读什么。
- 对视频和音频没有真正的阅读能力。
- 缺少事实冲突检测和主动交叉验证。
- 缺少过程记忆，容易重复尝试同样的搜索路径。
- 缺少“是否已经足够回答”的自我评估。

因此，结果往往像“搜索摘要”，不像“做过研究之后的回答”。

## 3. 本轮已明确的产品原则

根据当前对话，产品方向已经可以明确为：

- 不立刻搜索，先由模型做任务规划。
- 搜索不是单工具，而是一组带能力分工的工具箱。
- 第一阶段优先提取候选内容元数据，而不是直接全文抓取。
- 元数据至少包括标题、作者、发布时间、时长、播放量或阅读量。
- 候选内容先交给模型做价值判断和优先级排序。
- 只有少量高价值内容才进入深度阅读。
- 视频和音频必须能够被转成模型可读文本。
- 关键事实在必要时要主动做交叉验证。
- 系统需要有搜索过程记忆，避免重复绕圈。
- 每次工具调用后都需要有“继续搜还是已经能答”的反思环节。
- 每次任务完成后要沉淀可复用经验，包括平台、站点、工具和策略。

## 4. 核心愿景

这个产品不是“更强搜索”，而是“更像人类研究流程的搜索”。

理想工作方式如下：

1. 用户提出问题。
2. Planner 先拆解任务，决定应该优先查什么、去哪里查。
3. 系统调用不同工具做广度感知。
4. 模型根据候选元数据判断哪些内容值得深读。
5. 系统对高价值网页、视频、帖子做深度提取和正文化。
6. 对关键冲突信息进行交叉验证。
7. Evaluator 判断现有信息是否足够回答。
8. 如果不足，继续规划下一步；如果足够，组织证据并输出答案。
9. 最后沉淀本次搜索经验，供后续任务复用。

## 5. 系统总架构

建议将系统拆成五个核心层：

- Planner：负责任务理解、搜索计划和优先级制定。
- Action Space：负责真正的信息获取和处理。
- Scratchpad / Memory：负责记录中间事实、尝试路径和失败路径。
- Evaluator / Reflector：负责在每一步后判断信息是否足够，以及下一步动作。
- Experience Memory：负责沉淀这次任务的有效站点、搜索词、工具组合和失败模式。

### 5.1 多 Agent 编排架构

在实现层面，建议把上述五层能力封装成“Supervisor + Specialists”的多 Agent 系统，而不是让单个模型串行做完所有事。

这样做的原因是：

- 搜索、长文阅读、视频理解、事实核验本身就是不同类型的工作。
- 不同任务可以并行推进，减少整体时延。
- 高层规划和底层执行分离后，更容易监控质量和成本。
- 每个 Agent 可以拥有更稳定的提示词、工具权限和输出格式。

### 5.2 Supervisor / Manager Agent

Supervisor 是高层指挥官，负责全局编排。

核心职责：

- 理解用户原始问题
- 拆解任务和子问题
- 生成研究计划
- 决定要开启几条并行研究路径
- 把子任务分配给不同 Specialist Agent
- 监控各路径进度和结果质量
- 决定是否继续搜索、补查或停止
- 最终整合成统一回答

Supervisor 的关键价值：

- 把“搜什么、先搜什么、搜到什么时候停”从工具层提升到策略层。
- 根据任务复杂度动态决定串行还是并行。
- 避免所有 Agent 同时重复做同一类工作。

### 5.3 Specialist Agents

#### 5.3.1 Web Researcher Agent

专职使用 `enhanced_search`，负责广度感知和搜索路径扩展。

职责：

- 执行第一轮和后续轮次的结构化搜索
- 生成和迭代搜索词
- 优先抓取最新、权威、原始来源
- 汇总候选卡片，交回 Supervisor 做下一步分配

适合处理：

- 最新动态
- 官方公告
- 新闻、博客、论文、论坛入口发现

#### 5.3.2 Deep Analyst Agent

专职使用 `deep_read_page`，处理长文、报告、财报、论文、PDF 和图表内容。

职责：

- 提取干净正文
- 识别长文中的核心段落和关键论点
- 结合 Vision 能力理解图表、图片和页面截图
- 把复杂文档转成结构化分析结果

适合处理：

- 财报
- 技术博客
- 长篇文章
- PDF 报告
- 带图表的分析页面

#### 5.3.3 Multimedia Agent

专职使用 `extract_video_intel`，处理 YouTube、Bilibili、播客等多媒体内容。

职责：

- 提取视频或音频的字幕与转写
- 输出关键帧描述
- 生成按时间轴组织的摘要
- 标注关键观点出现的时间点

适合处理：

- 产品评测
- 开发者访谈
- 播客
- 教程和发布会

#### 5.3.4 Fact Verifier Agent

专职做交叉验证、冲突检测和可信度判断。

职责：

- 比较不同来源对同一事实的表述
- 检测数字、时间、引用的冲突
- 结合作者背景、发布时间、来源权威性做可信度打分
- 输出更可信结论和剩余不确定性

适合处理：

- 数据冲突
- 多来源不一致
- 真假混杂的热点事件

#### 5.3.5 Synthesizer Agent

负责把多个 Specialist 的输出整合成最终研究报告。

职责：

- 汇总各 Agent 的关键发现
- 组织证据链
- 标注置信度和争议点
- 输出结构化报告或最终答案

适合处理：

- 最终回答生成
- 深度研究报告
- 多来源比较总结

### 5.4 多 Agent 协作流程

建议协作方式如下：

1. Supervisor 接收用户问题并生成研究计划。
2. Supervisor 决定需要开启几条并行路径。
3. Web Researcher 先做广度感知，返回候选内容池。
4. Supervisor 根据候选类型把任务派发给 Deep Analyst、Multimedia 或 Fact Verifier。
5. 各 Specialist 将结构化结果写回 Scratchpad。
6. Evaluator 判断当前证据是否足够。
7. 如果不足，Supervisor 基于缺口发起新一轮任务。
8. 如果足够，Synthesizer 生成最终报告。

### 5.5 与现有五层能力的映射关系

- Planner 主要由 Supervisor 承担。
- Action Space 由各 Specialist Agent 调用。
- Scratchpad / Memory 作为所有 Agent 的共享工作区。
- Evaluator / Reflector 可以是独立 Agent，也可以作为 Supervisor 的强制步骤。
- Experience Memory 作为任务结束后的经验沉淀层，为后续 Supervisor 提供策略参考。

## 6. 系统主流程

为了便于产品、算法和工程团队对齐，当前系统可以抽象为一条标准主流程。

### 6.1 主流程总览

1. 用户提出问题。
2. Supervisor 先理解任务并拆解子问题。
3. Supervisor 制定研究计划，决定需要开启几条并行路径。
4. Web Researcher 先做广度感知搜索，返回候选内容卡片。
5. Supervisor 根据候选内容类型继续分发任务。
6. Deep Analyst、Multimedia、Fact Verifier 分别执行深度阅读、多媒体提取和交叉验证。
7. 各 Agent 将中间结果写入 Scratchpad。
8. Evaluator 判断当前信息是否足够回答原问题。
9. 如果不足，Supervisor 基于缺口继续发起下一轮任务。
10. 如果足够，Synthesizer 组织最终报告。
11. 任务完成后，系统把有效经验写入 Experience Memory。

### 6.2 分阶段说明

#### 第一阶段：任务理解与规划

这一阶段由 Supervisor 主导，目标不是立刻搜索，而是先明确：

- 用户到底要什么答案
- 问题包含哪些子问题
- 每个子问题需要什么类型的证据
- 哪些平台更值得优先搜索
- 哪些任务适合并行推进

这一阶段的产物通常包括：

- 任务目标
- 子问题列表
- 来源优先级
- 首轮搜索词
- 停止条件

#### 第二阶段：广度搜索与候选收集

这一阶段主要由 Web Researcher Agent 执行。

核心动作是调用 `enhanced_search`，快速拿到一批候选内容卡片，而不是立刻深读所有页面。

候选卡片重点包含：

- 标题
- 链接
- 平台
- 作者
- 发布时间
- 时长
- 播放量或阅读量
- 摘要
- 权威度信息

这一步的目标是建立一个“候选池”，供后续筛选和分发。

#### 第三阶段：候选筛选与任务分发

Supervisor 根据候选内容的类型、质量和相关性做下一步判断：

- 长文、财报、PDF、技术博客交给 Deep Analyst
- 视频、播客、访谈交给 Multimedia Agent
- 存在冲突或不一致的事实交给 Fact Verifier Agent

这一阶段的重点是：

- 不让所有结果都进入重处理
- 只把高价值候选送进深度分析
- 按内容类型选择最合适的 Specialist

#### 第四阶段：深度提取与正文化

这一阶段由各 Specialist Agent 执行。

典型情况如下：

- Deep Analyst 使用 `deep_read_page` 抽取干净正文、核心观点、图表信息
- Multimedia Agent 使用 `extract_video_intel` 提取字幕、转写、时间轴摘要、关键帧描述
- Fact Verifier 使用 `cross_check_facts` 比较冲突来源的时间、数字和引用依据

这一阶段的目标是把网页、视频、音频、帖子转成统一的模型可读证据。

#### 第五阶段：Scratchpad 记录与状态共享

所有 Agent 的中间结果都需要写回共享 Scratchpad。

这里记录：

- 已确认的事实
- 尚未解决的问题
- 已尝试过的搜索词
- 已读来源
- 已转写视频
- 冲突点
- 暂定结论

这样做的作用是：

- 避免重复搜索
- 避免重复阅读
- 让所有 Agent 基于同一状态协作

#### 第六阶段：评估与反思

每完成一轮搜索或提取后，Evaluator 都要判断：

- 信息是否足够回答原问题
- 哪些子问题已经解决
- 哪些子问题仍然缺失证据
- 当前结论是否缺少最新或权威来源
- 是否需要继续搜索

如果答案是不够，就回到 Supervisor 发起下一轮。

#### 第七阶段：整合输出

当 Evaluator 判断信息足够后，Synthesizer Agent 负责整合最终输出。

最终输出应尽量包含：

- 结论
- 证据链
- 关键来源
- 置信度
- 争议点
- 不确定性

必要时再由 Supervisor 做一次最终一致性检查。

#### 第八阶段：经验沉淀

任务完成后，系统需要总结这次搜索过程中的有效经验，写入 Experience Memory。

例如：

- 哪些平台适合这类问题
- 哪些搜索词命中率更高
- 哪些站点更常提供一手资料
- 哪些工具组合效率最好
- 哪些路径噪音过大应避免

### 6.3 一句话流程概括

当前产品的核心流程可以概括为：

> 拆题规划 -> 广度搜索 -> 候选筛选 -> 深度提取 -> 交叉验证 -> 评估是否足够 -> 整合输出 -> 沉淀经验

## 7. Planner：任务规划器

### 7.1 设计目标

用户提问后，系统不应立刻搜索，而应先由 Supervisor 主导任务规划。

Planner 需要先回答这些问题：

- 用户真正想知道的核心目标是什么？
- 问题里包含几个子问题？
- 每个子问题分别需要什么前置信息？
- 哪些来源更可能给出原始、可信、最新的内容？
- 哪个搜索方向应该优先？
- 先做广度感知，还是直接进入特定平台深搜？

### 7.2 Planner 输出格式

Planner 的输出建议结构化，包括：

- `task_goal`：本次任务最终要回答什么。
- `sub_questions`：拆解后的子问题列表。
- `required_evidence`：每个子问题需要的证据类型。
- `source_priority`：平台和来源的优先级列表。
- `initial_queries`：第一轮建议搜索词。
- `stop_condition`：什么条件下可以停止继续搜。

### 7.3 Planner 示例

用户问题：

> 苹果 2024 年发布的手机比 2023 年的在性能上提升了多少？

Planner 可以拆成：

- 子问题 1：2023 年和 2024 年对应机型分别是什么。
- 子问题 2：性能指标应该看什么，芯片、跑分、能效还是实际测试。
- 子问题 3：哪些来源最适合获取可信数据。

优先级示例：

1. 苹果官网和发布会材料。
2. 权威评测媒体和数据库。
3. 哔哩哔哩、YouTube、抖音数码区评测视频。
4. 社媒讨论和论坛用户实测。

## 8. Action Space：工具箱

系统不应只有单一 `search` 工具，而应提供一组面向不同阶段的工具。

### 8.1 广度感知工具：`enhanced_search(query, time_range?)`

模拟的人类思考：

> 我需要先看这个话题的主流内容是谁发的、什么时候发的、哪些来源更值得继续读。

输入：

- `query`
- `time_range`，可选，例如“过去一年”

返回字段建议：

- 标题
- URL
- 核心摘要
- 作者或来源媒体
- 发布时间
- 来源类型
- 来源权威度评级
- 平台

核心价值：

- 一次性拿到大量候选内容。
- 附带作者和时间，方便模型优先读最新或更权威的结果。
- 比传统网页列表更适合给模型做下一步规划。

### 8.2 深度阅读与降噪工具：`deep_read_page(url)`

模拟的人类思考：

> 这个页面看起来相关，但我只想看正文和核心观点，不想被广告、推荐和模板噪音干扰。

输入：

- `url`

返回字段建议：

- 清洗后的正文 Markdown
- 核心观点提取
- 作者
- 发布时间
- 引用来源
- 结构化段落

核心价值：

- 快速读取长文、报告、财报、技术博客。
- 让模型直接处理可读正文，而不是 HTML 噪音。

### 8.3 多模态信息萃取工具：`extract_video_intel(url)`

这是产品的核心增强点。

模拟的人类思考：

> 很多一手教程、评测、访谈和新闻现场都在视频里，我必须知道视频到底说了什么。

输入：

- 视频 `url`

返回字段建议：

- 视频标题
- 作者或频道
- 发布时间
- 时长
- 关键帧描述
- 完整字幕或转写文本
- 按时间轴整理的摘要
- 关键观点及对应时间点

核心价值：

- 把“20 分钟必须花 20 分钟看完”的人类约束变成几秒钟内可读的结构化文本。
- 把搜索对象从“网页”扩展到“流媒体内容”。
- 特别适合访谈、评测、演讲、教程、发布会、热点现场等场景。

### 8.4 信息交叉验证工具：`cross_check_facts(fact, source_a, source_b)`

模拟的人类思考：

> A 说 50%，B 说 30%，我要对比它们到底引用了什么。

输入：

- `fact`
- `source_a`
- `source_b`

返回字段建议：

- 双方结论
- 双方依据
- 时间差异
- 引用来源差异
- 更可信的一侧及原因
- 仍然存在的不确定性

核心价值：

- 当关键数据冲突时，系统不直接拍脑袋选一个答案。
- 让模型具备主动核验能力，提高最终回答的置信度。

### 8.5 后续可扩展工具

在第一版之外，后续还可扩展：

- `extract_social_thread(url)`：抓社媒主帖、转发链和高价值评论。
- `ocr_extract(url_or_image)`：提取图片、截图、海报中的文字。
- `timeline_builder(sources)`：自动生成事件传播时间线。
- `source_profile(url)`：建立作者、媒体、站点的可信度画像。

## 9. Scratchpad / Memory：记忆便签

人类做多步搜索时会在脑中保留上下文，系统也必须有中间记忆层。

### 9.1 需要记录的内容

- 已经确认的事实
- 仍然未确认的关键问题
- 已经尝试过的搜索词
- 哪些搜索词无效或噪音太大
- 哪些页面已经读过
- 哪些视频已经转写过
- 哪些来源互相矛盾
- 当前最可信的暂定结论

### 9.2 Scratchpad 的价值

- 避免重复搜索同样的问题。
- 避免模型死循环尝试同样的 query。
- 帮助 Evaluator 基于当前证据状态做判断。
- 为最后的证据组织提供过程记录。

## 10. Evaluator / Reflector：评估与反思器

系统每完成一次搜索、阅读、转写或核验后，都应强制进行一次反思。

### 10.1 反思问题

- 我现在掌握的信息是否足够回答原问题？
- 哪个子问题已经解决，哪个还没有？
- 当前答案是否缺少最新信息、权威来源或原始证据？
- 是否存在冲突事实尚未验证？
- 下一步最值得做的动作是什么？

### 10.2 Evaluator 输出

建议结构化返回：

- `is_sufficient`
- `resolved_questions`
- `missing_questions`
- `risk_notes`
- `next_best_action`
- `reason`

### 10.3 作用

- 强制系统避免“搜一点就答”。
- 也避免“信息已经足够却还在乱搜”。
- 让工具调用链变成一个闭环，而不是无约束展开。

## 11. Experience Memory：经验沉淀

每次任务完成后，系统应总结这次搜索过程中的有效经验。

### 11.1 记录内容

- 哪些站点在这类任务上最有价值
- 哪些平台适合找一手信息
- 哪些搜索词命中率高
- 哪些工具组合最有效
- 哪些来源经常产生冲突或噪音
- 哪些失败路径应在未来避免

### 11.2 价值

- 后续同类任务可以更快进入高价值搜索路径。
- 平台优先级可随着经验逐步动态调整。
- 帮助构建“任务类型 -> 搜索策略 -> 工具组合”的知识库。

## 12. 基于思考流程的完整工作流

以问题为例：

> Sora 模型现在的生成时长上限是多少？相比刚发布时有哪些技术架构上的更新？

### 12.1 Planner 阶段

拆解出两个子问题：

- 当前生成时长上限是多少。
- 相比刚发布时，技术架构有什么更新。

初步判断需要：

- 最新官方或权威资料
- 可能存在的开发者访谈
- 时间维度上的对照信息

Supervisor 在这一阶段决定至少开启两条并行路径：

- 路径 A：网页和官方资料路径
- 路径 B：访谈和视频资料路径

### 12.2 广度感知

调用：

- `enhanced_search("Sora maximum generation length current update")`
- `enhanced_search("Sora technical architecture update official interview")`

这一步主要由 Web Researcher Agent 执行。

系统拿到候选后，Supervisor 根据发布时间过滤掉过旧结果，优先关注最近的官方博客、访谈和技术说明。

### 12.3 深度阅读

对最新技术博客调用：

- `deep_read_page(url)`

这一步由 Deep Analyst Agent 执行。

系统提取出清洗正文和核心观点，得到部分技术更新信息。

### 12.4 多模态萃取

如果发现开发者访谈视频包含关键信息，则调用：

- `extract_video_intel(video_url)`

这一步由 Multimedia Agent 执行。

系统返回字幕和时间轴摘要，例如在特定时间点出现关于时长上限的新口述信息。

### 12.5 交叉验证

如果技术博客和访谈提到的能力范围存在差异，则调用：

- `cross_check_facts(fact, source_a, source_b)`

这一步由 Fact Verifier Agent 执行。

比较信息时间、上下文和引用来源，确认哪个版本更新、哪个更可靠。

### 12.6 Evaluator 判定

Evaluator 判断：

- 是否已经同时解决“当前时长上限”和“架构更新”两个子问题。
- 是否还缺失官方来源。
- 是否还需要继续搜索。

### 12.7 最终输出

输出不是简单拼接摘要，而是：

- 最终结论
- 所依据的网页与视频来源
- 时间线说明
- 技术更新点
- 不确定性说明

最终报告由 Synthesizer Agent 组织，再由 Supervisor 做最终一致性检查。

## 14. 工程落地路径

### 14.1 `enhanced_search`

可选实现路径：

- 接入 Exa.ai 这类为 LLM 设计的搜索 API
- 或自建“搜索聚合层 + 元数据标准化层”

目标不是只返回链接，而是返回带时间、作者、来源类型和权威度的结构化候选。

### 14.2 `deep_read_page`

可选实现路径：

- Firecrawl
- Jina Reader
- Playwright + Readability.js 自建正文抽取服务

目标是获得极干净的正文 Markdown 和结构化段落。

### 14.3 `extract_video_intel`

核心工程链路建议：

- YouTube：优先直接拉取现成字幕
- Bilibili 或无字幕视频：使用 `yt-dlp` 获取音频流
- 调用 OpenAI Whisper API 或本地 `Faster-Whisper` 做转写
- 再用小模型对长字幕做按时间戳摘要和观点提炼

目标输出：

- 完整转写文本
- 时间轴摘要
- 关键观点和对应时间点

### 14.4 `cross_check_facts`

初期可基于规则加模型联合实现：

- 对比时间
- 对比引用来源
- 对比数字是否来自同一上下文
- 输出差异和可信度判断

## 15. 高级能力边界

在当前“规划 -> 搜索 -> 深读 -> 核验 -> 综合”的基础上，产品还可以继续向更强的研究系统演化。以下能力不再只是“搜索增强”，而是把系统推进到“研究执行器”和“知识发现器”。

### 15.1 从只读走向读写执行

传统搜索系统本质上仍然是 Read-Only。

也就是说，它只能读取网页、视频、帖子和文档，然后基于文本进行判断。但人类在遇到关键结论时，往往不会止步于“别人怎么说”，而会继续做两件事：

- 自己动手验证
- 主动向外部世界发起询问

#### 15.1.1 动态代码验证

系统不应只提取结论，还应在必要时主动执行代码验证结论。

典型场景：

- 文章声称某组财务数据存在强相关
- 博客声称某个 benchmark 结果成立
- 媒体声称某产品过去几年指标持续改善

系统可以：

- 自动抓取原始 CSV、表格或网页数据
- 在沙盒环境中生成并运行 Python 脚本
- 调用 Pandas、NumPy、SciPy、statsmodels 等库做统计分析
- 输出复算结果、图表和误差说明

这类能力可以理解为：

> 从“基于文本信任”走向“基于数据重现和数学验证”

建议把这类能力抽象成：

- `run_verification_code(task, inputs)`

返回内容建议包括：

- 使用的数据源
- 自动生成的代码
- 执行结果
- 验证结论
- 是否支持原结论
- 误差或不确定性

#### 15.1.2 社会化探针

当系统发现公开互联网中不存在现成答案，或者问题太新、太专业时，它不应只能回答“不知道”。

系统可以进一步执行“社会化探针”：

- 自动起草邮件发给论文作者、研究者或公司联系人
- 在 Reddit、GitHub Issue、论坛、开发者社区发起提问
- 记录外部提问内容、时间和上下文
- 进入挂起状态，等待异步回复
- 一旦收到回复，自动唤醒并继续分析

这类能力本质上打破了“搜索必须几秒内同步完成”的限制，让系统具备异步研究能力。

建议把这类能力抽象成：

- `launch_social_probe(channel, prompt, context)`
- `resume_async_investigation(task_id, replies)`

返回内容建议包括：

- 探针目标平台
- 发出的提问文本
- 当前状态，已发送、等待中、已收到回复
- 回复摘要
- 回复可信度
- 基于回复更新后的结论

### 15.2 从使用工具走向创造工具

预设工具再强，也不可能覆盖所有网站和所有数据结构。

现实中会遇到：

- 交互式数据库网页
- 私有格式文件
- 奇怪的动态图表
- 需要特殊登录或点击流程的站点
- 无法直接复制或无公开 API 的页面

人类遇到这种情况时会自己写脚本，系统也应该具备类似能力。

#### 15.2.1 即时工具合成

当预设工具失败时，Agent 可以临时创建一个专用工具或脚本解决当前问题。

例如：

- 自动写 Playwright 脚本完成交互点击和数据提取
- 自动写网页解析器处理特殊 DOM 结构
- 自动写 Python 脚本清洗加密或嵌套数据
- 自动写图表解析代码提取 SVG、Canvas 或 JSON 数据

这种能力的本质是：

> 遇到新锁，当场配钥匙

建议把这类能力抽象成：

- `synthesize_tool(goal, target, constraints)`
- `run_ephemeral_tool(code, sandbox)`

返回内容建议包括：

- 工具生成目标
- 自动编写的脚本说明
- 执行日志
- 提取到的数据
- 工具是否成功完成任务
- 是否值得沉淀成长期工具

#### 15.2.2 与 Experience Memory 的关系

即时工具不应每次都“用完即焚”。

系统应记录：

- 哪类站点经常需要临时工具
- 哪段脚本值得抽象为正式工具
- 哪种失败模式未来应优先回避

这样，系统会逐步从“会临时写脚本”演化为“不断扩展自己的工具库”。

### 15.3 从检索已有知识走向暗知识发现

最高阶能力不再是搜到现成答案，而是在没有直接答案时提出新的假设和可能的解释链条。

这对应的是跨领域的隐藏关联发现，也可以理解为一种 Swanson Linking 能力。

典型形态是：

- 资料 A 证明了现象 X 和变量 Y 有关
- 资料 B 证明了变量 Y 与方法 Z 有关
- 但公开世界里还没有人把 X 和 Z 明确连起来

系统如果只会检索，就只能说“没有直接答案”。

但如果系统在后台持续构建跨来源、跨平台、跨模态的知识图谱，就有可能发现这种隐藏链条。

#### 15.3.1 Cross-Domain Synthesis

系统在抓取网页、视频、论文、专利和社区内容时，可以同步抽取：

- 实体
- 关系
- 时间
- 因果暗示
- 方法与结果
- 约束条件

再由知识图谱和推理层寻找：

- 跨领域共用变量
- 隐藏中介因素
- 未被直接写出的因果链
- 值得进一步验证的新假设

这类能力的输出不应该伪装成“已证实结论”，而应明确标记为：

- 推测
- 假设
- 待验证关联

建议把这类能力抽象成：

- `discover_hidden_links(topic, evidence_pool)`
- `propose_hypotheses(graph_state, constraints)`

返回内容建议包括：

- 发现的隐藏链条
- 关联所依赖的来源集合
- 推理路径
- 假设强度
- 推荐验证路径

#### 15.3.2 能力边界

这类能力不能直接等同于事实回答。

产品上必须明确区分三类输出：

- 已验证事实
- 高可信推断
- 新生成假设

否则系统会从“研究助手”滑向“会编故事的生成器”。

### 15.4 从瞬时搜索走向持续进化知识图谱

当前多数搜索系统即使接入 RAG，也只是把网页切成碎片后做向量检索，任务结束后上下文价值迅速衰减。

更高阶的形态应该是：

> Agent 不只返回答案，而是把每次搜索得到的事实、作者、时间、观点、数据和关系持续写入可演化的知识图谱。

#### 15.4.1 Self-Evolving Knowledge Graph

系统应将搜索结果拆解为节点和边，例如：

- 人物
- 机构
- 产品
- 事件
- 指标
- 观点
- 证据
- 时间关系
- 因果关系
- 利益关系

这样，系统沉淀的就不再是零散片段，而是可持续更新的研究知识底座。

#### 15.4.2 矛盾自动侦测

当新搜索结果和历史知识冲突时，系统应自动触发冲突解决策略。

例如：

- 昨天搜到某模型支持 1 分钟
- 今天搜到该模型支持 3 分钟

系统应自动执行：

- 标记冲突事实
- 对比发布时间和来源权威性
- 寻找第三方或官方来源补证
- 更新知识版本而不是直接覆盖旧值

#### 15.4.3 联想推理

图谱化后，系统可以主动关联跨任务知识。

例如在研究“低空经济”时，系统可以自动关联历史任务中已有的：

- 电池能量密度进展
- 无人机供应链变化
- 政策发布时间线
- 某城市试点范围扩张记录

在此基础上形成更强的推理，例如某地低空物流测试可能在下个季度扩容。

#### 15.4.4 技术实现方向

这类能力可考虑基于以下技术路线：

- GraphRAG
- 专家规则库
- 版本控制数据库
- 图谱推理引擎
- 冲突检测与版本仲裁模块

### 15.5 从公网搜索走向交互式探索

传统搜索只能读取搜索引擎能抓到的内容，但优秀研究员会主动创造获取路径。

更高阶的系统应具备：

- 交互能力
- 模拟能力
- 入驻式探索能力

#### 15.5.1 身份化探索

系统在合规边界内，可以根据任务场景采用不同“交互角色”获取信息。

例如：

- 以研究者身份在学术或开发者社区提问
- 以潜在客户身份与官网客服机器人沟通
- 以技术使用者身份进入论坛、文档站、工单系统或社区问答链路

重点不在伪装，而在于让系统从“读现成内容”升级到“主动制造新的信息输入”。

#### 15.5.2 Shadow APIs 主动探测

很多网页只是数据展示层，真正的数据来自前端背后的 JSON 或 GraphQL 接口。

系统应具备：

- 分析前端请求
- 发现隐藏接口
- 直接请求结构化数据
- 在页面展示不完整时获取更完整的数据包

这类能力可以显著提升数据质量，也比纯 DOM 抓取更稳定。

#### 15.5.3 深层公开资料探索

很多高价值信息并不在普通页面正文中，而在：

- 财报附件
- Excel 包
- GitHub 仓库
- 专利附录
- 技术文档下载区
- 登录后可见但合规公开的试看或试用内容

系统应具备跨附件、跨文件、跨入口的深层提取能力，而不是只停留在首页正文。

#### 15.5.4 合规边界

这类能力的边界必须非常明确。

产品上应明确禁止或严格限制：

- 绕过真实安全机制
- 非法突破登录、授权或付费限制
- 未经许可模拟真实身份进行高风险对外行为

否则系统会从研究工具变成高风险自动化代理。

### 15.6 从理解内容走向理解动机与博弈

更高阶的研究系统不只看“说了什么”，还要看：

- 谁在说
- 为什么现在说
- 他代表谁的利益
- 他可能忽略了什么
- 反方会如何攻击这个结论

#### 15.6.1 利益链分析

系统应对作者、媒体、机构和平台做背景建模。

例如：

- 文章是否来自利益相关方
- 媒体是否接受过相关公司投资
- 作者是否存在岗位、项目或商业绑定
- 内容发布时间是否与融资、发布会、政策节点重合

最终在报告中体现为：

- 可信度修正
- 偏见风险提示
- 利益关联说明

#### 15.6.2 多立场舆论与博弈建模

系统可同时跟踪不同语境下的观点：

- 中文社交媒体
- 英文技术论坛
- 欧洲政策简报
- 行业研究报告
- 开源社区讨论

然后对这些来源进行立场、动机、风险偏好和博弈关系建模，用于：

- 判断争议走向
- 预测舆论演化
- 预测市场或社区反应

#### 15.6.3 反向验证与 Red Teaming

当系统形成一个主要结论后，不应立即停止。

它应强制启动一条反方路径，专门搜索：

- 能推翻当前结论的证据
- 被忽视的反例
- 旧版本但仍有解释力的资料
- 与主流叙事相反的专家意见

这会显著降低系统因确认偏误而过早收敛的风险。

### 15.7 从信息检索走向自主闭环行动

最终极形态不是“我找到答案了”，而是：

> 搜索 -> 假设 -> 验证 -> 行动 -> 反馈 -> 再更新

#### 15.7.1 代码实验室

当系统搜索到某个算法、实现或 benchmark 结论时，可以自动在沙盒里复现它。

例如：

- 拉取代码
- 安装依赖
- 运行最小实验
- 对比文中宣称的性能指标
- 输出实际运行结果

此时用户获得的将不是“网页说了什么”，而是“系统实测到了什么”。

#### 15.7.2 实验设计

当公开资料不足以得出稳定结论时，系统可以进一步给出验证方案。

例如：

- 明确应该采集哪些数据
- 建议如何设计对照实验
- 自动写出测试脚本或分析脚本
- 给出执行步骤和预期观测指标

这让系统从“答案提供者”升级成“研究协作者”。

#### 15.7.3 主动监控与预警

系统不应只能被动等待提问。

如果用户存在长期研究主题，系统可对关键变量持续巡检，例如：

- 论文
- 视频
- 财报
- 开源代码库
- 专利
- 高价值账号动态
- 核心人物变动

当关键变量变化时，系统应结合历史图谱和既有结论自动发出影响分析。

#### 15.7.4 能力前提

要实现自主闭环行动，系统必须先具备：

- 长期状态管理
- 任务调度能力
- 稳定沙盒执行环境
- 严格的人类确认机制
- 清晰的权限边界

否则极易在成本、风险和误操作上失控。

## 16. MVP 建议

第一版不要试图做全平台、全能力，而是先验证最核心闭环。

### 16.1 MVP 必须具备的能力

- 有 Planner，而不是直接搜
- 有 Supervisor 做任务编排
- 有广度感知搜索
- 有网页深读
- 有视频转文本和时间轴摘要
- 有 Scratchpad 记录过程
- 有 Evaluator 判断是否继续
- 能输出证据化答案

### 16.2 MVP 的 Agent 组合建议

第一版不需要把所有 Specialist 都做得很重，建议先实现：

- Supervisor / Manager Agent
- Web Researcher Agent
- Deep Analyst Agent
- Multimedia Agent
- Synthesizer Agent

Fact Verifier Agent 在 MVP 中可以先做成一个轻量核验模块，第二阶段再升级成独立 Agent。

### 16.3 建议的 MVP 平台组合

- 通用网页
- 哔哩哔哩
- YouTube 或抖音
- 一个社媒平台，如微博或 Reddit

### 16.4 MVP 先不做或弱化

- 全平台评论区深挖
- 复杂的作者可信度长期画像
- 大规模用户界面产品化
- 全自动事件图谱构建
- 自动对外发帖、发邮件的社会化探针
- 通用的即时工具合成平台
- 大规模暗知识发现和假设生成
- 持续进化知识图谱的全量自动更新
- 身份化探索和 Shadow API 主动探测
- 大规模偏见建模与舆论场预测
- 长周期的主动监控和预警系统

## 17. 阶段路线图

为了避免产品目标不断膨胀、工程范围失控，建议把整体演进拆成四个阶段：

- MVP
- Phase 2
- Phase 3
- Ultimate

### 17.1 MVP：先验证基础研究闭环

阶段目标：

- 证明“先规划、再搜索、再深读、再输出证据化答案”这条主链路成立。
- 证明视频和网页深读相比传统 Web Search 有明显增益。
- 证明多 Agent 编排不会把系统复杂度推到不可控。

核心能力：

- Supervisor + 基础 Specialist Agent 编排
- `enhanced_search`
- `deep_read_page`
- `extract_video_intel`
- 基础 `cross_check_facts`
- Scratchpad / Evaluator
- 证据化回答输出

典型交付：

- 面向开发者的 API 或内部研究控制台
- 支持 3 到 4 个重点平台
- 支持快速问答和深度研究摘要

阶段完成标准：

- 用户明确感知到结果优于普通 Web Search
- 视频转文本和网页深读真正进入主链路
- 单次任务成本、时延和错误率基本可控

### 17.2 Phase 2：把系统升级为可验证研究工具

阶段目标：

- 从“会读内容”升级到“会验证内容”。
- 从一次性任务升级到轻量长期记忆。
- 从单次问答升级到更稳定的研究工作台。

核心能力：

- 动态代码验证
- 轻量版本化知识图谱
- 冲突检测与冲突仲裁
- 更强的 Fact Verifier
- 附件、表格、PDF、财报等深层资料提取
- Red Teaming 式反向验证

典型交付：

- 数据复算报告
- 带版本差异的事实结论
- 简易研究项目工作区

阶段完成标准：

- 系统能对关键数据结论做复算，而不是只复述
- 新旧知识冲突时，系统能保留版本并给出解释
- 复杂资料如 PDF、表格、财报附件的处理质量稳定

进入本阶段的前提：

- MVP 的搜索闭环稳定
- 核心平台抽取成功率足够高
- 已有基础沙盒和执行环境

### 17.3 Phase 3：把系统升级为持续研究与策略分析平台

阶段目标：

- 从“回答问题”升级到“维护研究主题”。
- 从“理解内容”升级到“理解动机、偏见和博弈”。
- 从单次任务型产品升级到主题跟踪型产品。

核心能力：

- Self-Evolving KG 的持续更新
- 利益链分析
- 多立场舆论与博弈建模
- 长期主题跟踪
- 主动监控与预警
- 更成熟的 Experience Memory

典型交付：

- 长期研究档案
- 主题时间线
- 舆论场和立场差异分析
- 关键事件预警与影响分析

阶段完成标准：

- 系统能在长期运行中保持低噪音、高信号
- 用户能明显感知“它记住了过去研究过什么”
- 结论不只回答事实，还能解释动机和风险

进入本阶段的前提：

- Phase 2 的知识版本管理稳定
- 可信度和偏见建模已有基本框架
- 有稳定的任务调度和长期状态存储

### 17.4 Ultimate：把系统升级为研究执行器和知识发生器

阶段目标：

- 从“研究助手”升级到“研究执行器”。
- 从“检索已有答案”升级到“提出新假设和新验证路径”。
- 从同步问答升级到异步、持续、自主的研究闭环。

核心能力：

- 社会化探针
- 即时工具合成
- Shadow API 探测和更深层的公开信息探索
- Cross-Domain Synthesis
- 假设生成与推荐验证路径
- 搜索 -> 假设 -> 验证 -> 行动 -> 反馈 的闭环

典型交付：

- 自动唤醒的异步研究任务
- 针对未知问题的验证方案
- 跨领域隐藏关联发现报告
- 高级探索模式下的新假设输出

阶段完成标准：

- 系统不仅能总结世界已有内容，还能提出值得验证的新方向
- 系统能够稳定管理高风险能力，不出现失控调用
- 用户把它视为“研究系统”，而不是“搜索插件”

进入本阶段的前提：

- Phase 3 的长期状态、知识图谱和预警体系稳定
- 人类确认和权限管理机制成熟
- 合规、风控、审计机制完整

### 17.5 路线图原则

这条路线图有几个原则必须坚持：

- 每个阶段都必须先证明价值，再扩能力边界。
- 先做高频、刚需、确定性的能力，再做高风险和探索性能力。
- 所有“对外动作”和“自动执行动作”都必须晚于基础搜索闭环。
- 假设生成类能力必须晚于事实验证能力。
- 长期记忆和图谱必须在基础抽取质量稳定后再放大投入。

## 18. 成功标准

这个产品是否成立，核心看以下指标：

- 用户是否明显感觉它比普通 Web Search 更像“研究助手”。
- 系统是否真的先规划再行动，而不是直接检索。
- 候选内容筛选后，深度阅读命中率是否更高。
- 视频类内容是否有效转成可用证据。
- 关键事实冲突时，系统是否能主动核验。
- 最终答案质量是否明显优于“只看网页摘要”。
- 单次任务时延和成本是否在可控范围内。
- 多 Agent 并行后，整体研究效率是否明显提升。
- 在进入高级阶段后，系统是否能够稳定完成代码验证而不是只复述分析。
- 在图谱阶段后，系统是否能稳定处理知识冲突并保留版本演化记录。
- 在长期运行阶段后，系统是否能持续输出高价值预警而不是大量噪音。

## 19. 主要风险与 trade-off

- 平台接入越多，维护成本越高。
- 视频转写越充分，时延和费用越高。
- 评论区价值高，但噪音极大。
- 热度高不等于信息密度高。
- 来源权威度评级本身也需要持续校正。
- Planner 如果过度复杂，会拖慢整体链路。
- Scratchpad 如果设计过重，会让系统状态难管理。
- 自动执行代码会带来沙盒安全、资源控制和结果可复现问题。
- 自动对外发帖或发邮件会带来合规、品牌和滥用风险。
- 暗知识发现如果没有严格标注，很容易和事实回答混淆。
- 知识图谱持续演化会带来版本膨胀、冲突管理和维护成本问题。
- 身份化探索、主动探测接口会触及更敏感的法律与平台政策边界。
- 偏见建模如果做得粗糙，容易把复杂动机简化成武断标签。
- 主动监控和预警如果阈值设计不当，会造成高噪音和用户疲劳。

## 20. 下一轮最值得继续确认的问题

1. 第一版优先支持哪 4 个平台？
2. 第一版先做面向开发者的 API，还是先做可直接体验的产品界面？
3. `authority_score` 应该如何定义，按站点、作者还是内容类型打分？
4. `extract_video_intel` 第一版是否只支持 YouTube 和 B 站？
5. 评论区抽取在 MVP 里是否必须纳入？
6. Planner 输出格式是否要固定成标准 JSON，便于后续 agent 编排？
7. 动态代码验证是否要在第二阶段引入，而不是放到 MVP？
8. 社会化探针是否只允许人工确认后再真正对外发送？
9. 假设生成类输出是否要单独放在“探索模式”中，避免与事实答案混用？
10. Self-Evolving KG 是从第二阶段开始做轻量版，还是等基础搜索闭环稳定后再做？
11. Shadow API 探测和身份化探索是否完全限制在白名单站点内？
12. 主动监控和预警是否要作为独立产品模式，而不是默认能力？
